Compaction can now fold a single turn that outgrew the trigger on its own. Both fold cuts were `HumanMessage`-boundary-only, so a long tool loop under one user message (60–100+ ai/tool cycles — the exact shape agentic workloads make) had no boundary anywhere in the foldable region: the trigger fired every cycle, the planner returned null every cycle, and context rode at peak for the rest of the turn (field incident: 5–6+ consecutive cycles at 114–122k input). `summarize`-strategy folds now fall back to cutting at the newest `AIMessage` at/before the naive cut — safe on the wire, since tool results always follow their `tool_calls` parent, an AI-opening retained span never strands one, and the pinned head still opens the history. Because this is the only cut that folds the running turn's own ask, it is summarize-only (the summary stands in for the span): `drop` folds decline it, and a summarize failure on such a fold declines rather than drop-falling-back, so the model never loses its current instruction without a record. Two guards ride along: the fallback requires a resolvable `ContextWindowMiddleware` (the only thing that renders the summary into the model's view — standalone `provideCompaction` + summarize would erase the ask into a channel nothing reads; use `provideManagedContext`), and after 3 consecutive summarize failures on such folds a thread stops attempting them, since each attempt sends the whole peak-context span to the summarizer.
